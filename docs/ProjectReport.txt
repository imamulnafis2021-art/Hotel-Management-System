The Starlight Hotel Management System is a comprehensive,
production-quality C++17 console application designed to
automate and streamline all operations of a modern hotel.
The system implements all four pillars of OOP: Encapsulation,
Inheritance, Polymorphism, and Abstraction. It supports
role-based authentication (Admin, Receptionist, Manager),
persistent file-based storage for all data entities, STL-
based searching and sorting, automated invoice generation,
smart room recommendations, and an advanced analytics
dashboard. The system was upgraded from a procedural design
to a fully object-oriented multi-file architecture following
SOLID principles.

1. INTRODUCTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1.1 Background
Hotels are complex businesses involving room reservations,
guest management, food and beverage, housekeeping, transport,
billing, and staff management. Manual processes are error-
prone and slow. A Hotel Management System (HMS) automates
these processes, improves accuracy, and enhances guest
satisfaction.

1.2 Problem Statement
The original system was a single-file procedural C++ program.
While functional, it lacked proper OOP design, had global
variables everywhere, no inheritance hierarchy, no persistent
storage for most data types, no authentication, and no
advanced analytics. This project upgrades it to a full OOP
design in 14 phases.

1.3 Objectives
- Convert to multi-file OOP architecture
- Implement Inheritance and Polymorphism
- Add full file persistence for all entities
- Add role-based authentication (3 roles)
- Add Search and Sort using STL
- Generate invoices automatically
- Add smart room recommendations
- Add advanced analytics dashboard
- Replace weak validation with regex
- Use modern C++17 date/chrono utilities

1.4 Scope
The system covers: Room Management, Booking, Checkout,
Food Service, Laundry, Transport, Complaint Management,
Staff Management, Payments, Revenue Reports, Dashboard,
and File Persistence.

exercise.

2.2 Object-Oriented Programming in C++
Bjarne Stroustrup's C++ (The C++ Programming Language, 4th Ed)
defines the four pillars used in this project:
- Encapsulation: Private data with public accessors
- Inheritance: Person -> Customer, Person -> Staff
- Polymorphism: virtual displayInfo() in base Person
- Abstraction: Pure virtual functions, abstract base class

2.3 SOLID Principles Applied
S - Single Responsibility: Each class handles one concern
O - Open/Closed: New room types can be added without changing
    existing booking logic
L - Liskov Substitution: Customer/Staff usable as Person*
I - Interface Segregation: Separate manager/receptionist menus
D - Dependency Inversion: FileManager is standalone utility

4.2 Phase-by-Phase Implementation

PHASE 1: OOP Refactoring
  - Created HotelManagementSystem as the main orchestrator
  - All global functions converted to class methods
  - All global vectors encapsulated as private members
  - Global counters encapsulated with proper initialization

PHASE 2: Inheritance
  - Created abstract base class Person with: name, phone,
    address, email fields and getters/setters
  - Customer derives from Person (adds booking-specific fields)
  - Staff derives from Person (adds employment fields)

PHASE 3: Polymorphism
  - Person declares pure virtual: virtual void displayInfo() = 0
  - Customer implements displayInfo() showing booking summary
  - Staff implements displayInfo() showing employment summary
  - Runtime polymorphism demonstrated via Person* pointers

PHASE 4: File Persistence
  - FileManager handles save/load for ALL 8 entity types
  - Custom escape encoding handles newlines in text fields
  - Counter persistence ensures IDs never repeat across sessions

PHASE 5: Login System
  - credentials.txt auto-created with 3 default accounts
  - Roles: Admin (full access), Receptionist (front desk),
    Manager (reports + staff)
  - 3 login attempts before lockout

PHASE 6: Search System
  - searchRoom(): by ID, type, price range
  - searchCustomer(): by name, booking ID, room ID
  - searchStaff(): by name, ID, position
  - All use STL find_if and string::find

PHASE 7: Sorting System
  - sortRooms(): price asc/desc, rating, ID
  - sortCustomers(): name, booking ID, total bill
  - sortStaff(): name, salary, join date
  - All use STL sort() with lambda comparators

PHASE 8: Invoice Generation
  - generateInvoice() in FileManager
  - Creates invoice_XXXX.txt in data/ folder
  - Includes all charges: room, food, laundry, transport
  - Generated on booking and on checkout

PHASE 9: Validation
  - validateEmail() uses std::regex
  - validatePhone() checks digit count and characters
  - validateNID() checks length and digit content
  - validateName() checks alphabetic characters
  - All inputs wrapped in do-while with error messages

PHASE 10: Date Management
  - Using <ctime> with strftime/mktime for YYYY-MM-DD format
  - calculateCheckOutDate() adds days using mktime arithmetic
  - daysBetween() computes difference between two dates

PHASE 11: Date-Based Room Availability
  - Room stores vector<pair<string,string>> of booking slots
  - isAvailableForDates() checks for date range overlap
  - Prevents double-booking on same room for overlapping dates

PHASE 12: Advanced Dashboard
  - Occupancy Rate (%)
  - Average Stay Duration (days)
  - Most Popular Room Type (by booking count)
  - Top Revenue Room
  - Top Spending Customer
  - Monthly Revenue Summary (YYYY-MM aggregation)
  - Complaint Priority Statistics

PHASE 13: Smart Room Recommendation
  - Input: budget, number of guests, preferred type
  - Filters: price <= budget, capacity >= guests, type match
  - Outputs matching rooms with WHY THIS ROOM explanation

PHASE 14: Multi-File Architecture
  - 8 header files + 7 source files
  - Clear separation of concerns
  - Makefile for single-command compilation

4.3 Key Design Decisions

ID Counter Persistence: Counters saved to counters.txt so
IDs are unique across sessions (no ID collision on restart).

Escape Encoding: Multi-line text fields (amenities,
complaints, special requests) use \n escape to survive
line-by-line file I/O without corruption.

STL Throughout: vector, map, algorithm (sort, find_if,
count_if, remove_if) used instead of raw arrays/loops.

const Correctness: Read-only methods marked const,
parameters passed as const reference where appropriate.

6. OOP CONCEPTS DEMONSTRATED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ENCAPSULATION
  - All data members private in Room, Customer, Staff
  - Access only via getters/setters
  - Business logic in class methods, not free functions

INHERITANCE (Phase 2)
  - Person (base) -> Customer (derived)
  - Person (base) -> Staff (derived)
  - Customer and Staff inherit: name, phone, address, email
  - Constructors chain via : Person() initializer

POLYMORPHISM (Phase 3)
  - Person declares pure virtual void displayInfo() = 0
  - Customer::displayInfo() shows guest booking info
  - Staff::displayInfo() shows employment info
  - Runtime polymorphism: Person* p = new Customer();
    p->displayInfo(); // calls Customer's version
  - Person is an abstract class (cannot be instantiated)

ABSTRACTION
  - Person hides implementation details from callers
  - FileManager hides file I/O complexity
  - Utility namespace hides date/validation complexity

STL USAGE
  - std::vector for all collections
  - std::sort with lambda comparators (Phases 7)
  - std::find_if for searching (Phase 6)
  - std::count_if for statistics
  - std::remove_if + erase for deletion
  - std::map for revenue aggregation (Phase 12)
  - std::regex for email validation (Phase 9)
  - std::filesystem for directory creation (Phase 4)

EXCEPTION HANDLING
  - try/catch around stod/stoi conversions
  - Input validation loops prevent bad state
  - File I/O checked with if(!stream) guards

MODERN C++17 FEATURES
  - std::filesystem::create_directories
  - Inline variables in Color namespace
  - Structured bindings in map iteration
  - std::get_time / std::put_time
  - Lambda expressions throughout

8. BUG FIXES FROM ORIGINAL CODE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BUG-01: Double cin.get() in main menu caused Enter skip
  Fix: Replaced with clearInput() after every cin >> input

BUG-02: getCurrentDate() returned full ctime string
  Fix: Used strftime to return clean "YYYY-MM-DD" format

BUG-03: validatePhone() only accepted exactly 10 or 11 digits
  Fix: Now accepts 10-15 digits, handles international formats

BUG-04: No persistence for customers/staff/payments/orders
  Fix: Complete file persistence added for all 8 entity types

BUG-05: IDs reset to 1001 on every restart (ID collision)
  Fix: Counter persistence in counters.txt

BUG-06: Global variables across entire file (unsafe)
  Fix: All encapsulated in HotelManagementSystem class

BUG-07: No checkout invoice saved to file
  Fix: FileManager::generateInvoice() called at checkout

BUG-08: validateEmail() was too permissive (no regex)
  Fix: std::regex pattern used for proper RFC-style check

BUG-09: Complaint registration had no category or priority
  Fix: Added category (7 types) and priority (1-3) fields

BUG-10: Revenue counters not recalculated on load
  Fix: totalRevenue and pendingRevenue recalculated in loadAll()
